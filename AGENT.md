# AGENT.md

## 目标

这个文件用于指导 AI 收集、整理和维护 **Pi 扩展（Pi Extensions）/ Skills / 相关仓库** 资料。

当用户提供一个 GitHub 仓库链接时，AI 需要读取该仓库的公开信息与代码结构，并将其整理为一份结构化记录，存放到**对应分类文件夹**中。

这个仓库的核心目的不是简单收藏链接，而是建立一个**可检索、可比较、可持续追加**的资料库。

---

## 当前仓库组织方式

资料库采用“**分类文件夹 + 仓库独立目录**”的结构：

- 根目录 `README.md`：只作为导航页 / 总目录
- 每个主分类一个文件夹
- 每个仓库在其分类文件夹下有一个独立目录
- 仓库的详细记录写入该目录下的 `README.md`

### 分类目录

- `Command/`
- `Tool/`
- `Event-Hook/`
- `UI-Notification/`
- `Workflow-Automation/`
- `Integration/`
- `Template-Example/`
- `Utility-Developer-Experience/`

### 单条记录的存放方式

例如某仓库被归类为 `Integration`：

- 分类目录：`Integration/`
- 仓库目录：`Integration/<repo-name>/`
- 详细记录：`Integration/<repo-name>/README.md`

---

## 你的职责

当用户发送一个 GitHub 链接时，你需要完成以下任务：

1. 识别这个仓库是否与 **Pi 扩展 / Skills / agent 能力扩展** 相关。
2. 阅读仓库的 README、目录结构、关键源码文件、配置说明。
3. 判断它属于哪一种主分类。
4. 用清晰简洁的中文整理出一份结构化记录。
5. 将详细记录写入对应分类目录下的仓库文件夹中。
6. 更新根目录 `README.md`，加入可点击的导航链接。
7. 如有必要，更新对应分类目录下的 `README.md`，补充该仓库入口。
8. 尽量提炼这个仓库的实际价值，而不是只重复 README。

---

## 处理原则

### 1. 优先查看这些内容

分析仓库时，优先看：

- `README.md`
- `package.json`
- `tsconfig.json`
- `src/` 目录
- 扩展入口文件
- 与 command / tool / event / hook / ui 相关的代码
- 安装方法
- 配置方法
- 使用示例

### 2. 不要只照抄 README

你的工作是“整理与解释”，不是机械搬运。

应该尽量回答：

- 这个仓库是做什么的？
- 它解决了什么问题？
- 它主要通过什么机制工作？
- 它更适合什么场景？
- 它有没有明显的特点、限制或依赖？

### 3. 尽量按“功能”分类，而不是按宣传语分类

优先依据实际实现与使用方式判断主分类，而不是照搬作者自己的 marketing 语言。

### 4. 信息不确定时要明确说明

不要编造。请使用：

- 未明确说明
- 暂未发现
- 需要进一步阅读源码确认
- 当前判断主要基于 README / 目录结构

## 分类体系

优先从下列类别中选择一个**主分类**；必要时可附加 1 到 2 个标签。

### 主分类

1. **Command 扩展**
   - 注册命令
   - 提供快捷调用
   - 通过命令触发操作

2. **Tool 扩展**
   - 注册工具
   - 提供给 AI 调用的能力
   - 返回结构化结果或执行任务

3. **Event / Hook 扩展**
   - 监听 session_start、tool_call 等事件
   - 在生命周期节点自动执行逻辑

4. **UI / Notification 扩展**
   - 与界面提示、通知、反馈展示相关

5. **Workflow / Automation 扩展**
   - 自动化流程
   - 多步骤串联
   - 工作流编排

6. **Integration 扩展**
   - 对接第三方服务、API、数据库、本地工具
   - 如 GitHub、Slack、CLI、文件系统等

7. **Template / Example 扩展**
   - 主要用于演示 Pi 扩展写法
   - 更偏学习样板或二次开发参考

8. **Utility / Developer Experience 扩展**
   - 辅助开发
   - 提升调试、日志、开发体验
   - 通用技能包 / 工具集合 / 开发辅助项目

### 可选标签

- `beginner-friendly`
- `example`
- `production-oriented`
- `requires-config`
- `local-only`
- `third-party-api`
- `typescript`
- `minimal`
- `experimental`
- `skill-collection`
- `non-extension`

## 根目录 README 的职责

根目录 `README.md` **不保存长篇详细记录**，只承担以下作用：

1. 简要说明这个仓库是做什么的
2. 列出所有分类目录
3. 提供当前已收录仓库的导航链接
4. 让用户可以点击进入对应分类或具体仓库目录

### 根目录 README 应保持的特点

- 简短
- 清晰
- 可点击导航
- 不堆积详细分析

---

## 分类目录 README 的职责

每个分类目录下的 `README.md` 用来：

1. 说明这个分类收录什么类型的项目
2. 列出当前该分类下已收录的仓库
3. 链接到各仓库独立目录

---

## 去重规则

当用户再次提供链接时：

1. 先检查该仓库是否已经存在于某个分类目录下
2. 如果已经存在：
   - 不重复新增目录
   - 更新原有 `README.md`
   - 必要时补充新的发现
3. 如果是同一个仓库的不同页面 / 不同分支：
   - 归并到同一仓库目录中
   - 必要时在备注中说明差异

---

## 分类判断优先级

如果一个仓库同时具备多个特征，按以下优先级确定主分类：

1. 它最核心的能力是什么
2. 用户最可能因为什么用途去使用它
3. 仓库代码中占比最大的实现部分是什么

例如：

- 既有 command 又有 tool，但核心是给 AI 提供调用能力，则主分类选 **Tool 扩展**
- 主要用于学习示例，即使也注册了 command，仍可归为 **Template / Example 扩展**
- 主要监听事件并自动处理逻辑，则归为 **Event / Hook 扩展**
- 如果它本质上是一个 skills 集合，而不是标准 Pi extension，可归为 **Utility / Developer Experience 扩展**，并加标签 `skill-collection`、`non-extension`

---

## 当信息不足时的处理方式

如果仓库内容过少、README 很弱、代码不完整，输出时要明确写：

- 仓库信息较少
- 当前判断主要基于目录结构或少量源码
- 结论可能需要后续验证

不要编造不存在的安装步骤、配置项或功能。

---

## 用户发来链接后的实际执行要求

当用户给出 GitHub 链接时，你需要：

1. 分析链接对应的仓库
2. 判断主分类
3. 在对应分类目录下创建仓库文件夹
4. 将详细分析写入该仓库目录下的 `README.md`
5. 更新根目录 `README.md` 导航
6. 如有必要，更新分类目录 `README.md`
7. 保留原始链接
8. 用中文写说明
9. 如有必要，补充“为什么值得收集”

## 简版输出格式（当用户只想快速看结果时）

如果用户只需要快速结果，可以输出：

- 名称
- 链接
- 分类
- 一句话说明
- 详细记录所在路径
- 是否值得收集

但默认仍应优先维护完整的仓库记录文件。

- 可点击导航
- 可快速理解
- 可用于后续开发参考
