# AGENT.md

## 目标

这个文件用于指导 AI 收集、整理和维护 **Pi 扩展（Pi Extensions）** 相关资料。

当用户提供一个 GitHub 仓库链接时，AI 需要读取该仓库的公开信息与代码结构，并将其整理为一份结构化的扩展记录，追加到本知识文档中。

这个文档的核心目的不是简单收藏链接，而是建立一个**可检索、可比较、可复用**的 Pi 扩展资料库。

---

## 你的职责

当用户发送一个 GitHub 链接时，你需要完成以下任务：

1. 识别这个仓库是否与 **Pi 扩展** 相关。
2. 阅读仓库的 README、目录结构、关键源码文件、配置说明。
3. 判断它属于哪一种扩展类型。
4. 用清晰简洁的语言写出说明。
5. 把仓库链接保存在文档里。
6. 尽量提炼出这个扩展的实际价值，而不是只重复 README。
7. 按统一格式输出，便于后续持续追加和检索。

---

## 处理原则

### 1. 优先看这些内容
当分析一个 GitHub 仓库时，优先查看：

- README.md
- package.json
- tsconfig.json
- src/ 目录
- extension 入口文件
- 与 command / tool / event / hook / ui 相关的代码
- 安装方法
- 配置方法
- 使用示例

### 2. 不要只照抄 README
你需要做的是“整理与解释”，不是机械搬运。

应该回答这些问题：

- 这个扩展是做什么的？
- 它解决了什么问题？
- 它主要通过什么机制工作？
- 它更适合什么场景？
- 它有没有明显的特点、限制或依赖？

### 3. 尽量从“功能”而不是“宣传语”来分类
分类时优先依据实际功能，而不是仓库作者自己写的 marketing 语言。

### 4. 信息不确定时要明确说明
如果某项信息无法确认，不要编造。请明确写：

- 未明确说明
- 暂未发现
- 需要进一步阅读源码确认

## 分类体系

当你整理 Pi 扩展时，优先从下列类别中选择一个主分类；必要时可加 1 到 2 个副标签。

### 主分类

1. **Command 扩展**
   - 注册命令
   - 提供快捷调用
   - 通过命令触发操作

2. **Tool 扩展**
   - 注册工具
   - 提供给 AI 调用的能力
   - 例如生成文本、执行任务、返回结构化结果

3. **Event / Hook 扩展**
   - 监听 session_start、tool_call 等事件
   - 在特定生命周期自动执行逻辑

4. **UI / Notification 扩展**
   - 与界面提示、通知、反馈展示相关
   - 例如 notify、状态提示、交互反馈

5. **Workflow / Automation 扩展**
   - 用于自动化流程
   - 将多个动作串起来执行
   - 偏向工作流编排

6. **Integration 扩展**
   - 对接第三方服务、API、数据库、本地工具
   - 例如 GitHub、Slack、文件系统、命令行工具

7. **Template / Example 扩展**
   - 主要用于演示 Pi 扩展写法
   - 适合学习和二次开发

8. **Utility / Developer Experience 扩展**
   - 辅助开发
   - 提升调试、日志、开发体验、结构组织等

### 可选标签

可以附加这些标签：

- `beginner-friendly`
- `example`
- `production-oriented`
- `requires-config`
- `local-only`
- `third-party-api`
- `typescript`
- `minimal`
- `experimental`

- 功能说明
- 适用场景
- 核心机制

- 个人备注（如果用户要求加入主观判断）

如果某些字段无法确认，可以写“未明确说明”。

## 输出风格要求

写作要求：

- 使用中文
- 简洁、准确、可检索
- 不说空话
- 不堆砌宣传语
- 不要过度展开无关细节
- 优先说明“它能做什么”和“怎么接入”
- 尽量让人一眼就知道是否值得收藏

## 去重规则

当用户再次提供链接时：

1. 先检查文档中是否已经存在这个仓库。
2. 如果已经存在：
   - 不要重复新增
   - 改为更新说明
   - 可以补充新的发现
3. 如果是同一个仓库的不同分支或不同页面：
   - 归并到同一条记录中
   - 必要时在备注中说明差异

---

## 分类判断优先级

如果一个扩展同时具备多个特征，请按以下优先级确定主分类：

1. 它最核心的能力是什么
2. 用户最可能因为哪种用途去使用它
3. 仓库代码中占比最大的实现部分是什么

例如：

- 既有 command 又有 tool，但核心是给 AI 提供调用能力，则主分类选 **Tool 扩展**
- 主要用于学习示例，即使也注册了 command，仍可归为 **Template / Example 扩展**
- 主要监听事件并自动处理逻辑，则归为 **Event / Hook 扩展**

---

## 当信息不足时的处理方式

如果仓库内容过少、README 很弱、代码不完整，输出时要明确写：

- 仓库信息较少
- 当前判断主要基于目录结构或少量源码
- 结论可能需要后续验证

不要编造不存在的安装步骤、配置项或功能。

---

## 用户发来链接后的实际执行要求

当用户给出一个 GitHub 链接时，你需要：

1. 分析链接对应的仓库
2. 按模板产出一条完整记录
3. 将记录放入“Pi Extensions Collection”文档格式中
4. 保留原始链接
5. 自动分类
6. 用中文写说明
7. 如有必要，补充“为什么值得收集”

---

## 建议的简版输出格式（当用户只想快速看结果时）

如果用户只需要简短结果，可以输出：

- **名称**
- **链接**
- **分类**
- **一句话说明**
- **是否值得收集**
- **简短备注**

但默认仍应优先使用完整模板。

## 最终目标

你不是在做一个“GitHub 链接收藏夹”，而是在帮用户建立一个：

- 可持续积累
- 可分类检索
- 可快速理解
- 可用于后续开发参考

的 **Pi 扩展知识库**。
